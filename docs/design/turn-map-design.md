# 턴(#151) · 맵/통신(#180) 완전 설계

작성: Opus 4.8 인라인 설계(멀티에이전트 울트라코드는 헤드리스 자율 실행에서 권한 스트림 미동작으로 미사용).
목적: 사용자가 확정한 턴/맵 개념을 현행 코드 기준으로 구현 가능한 설계로 정리. **구현은 승인 후** — §6 위험부분 참고.

---

## 0. 요약

두 시스템을 함께 설계한다.

- **턴(#151)**: 고정 "전원 1턴" 방식을 폐기하고 **행동별 소요시간(분) 기반 비동기 진행**으로. 각 플레이어는 자기 행동이 끝나는 시각(`busyUntil`)까지 "행동 중"이고, GM이 행동마다 소요시간을 매긴다. 시계가 가장 이른 `busyUntil`까지 흐르고, 그 시점에 자유로워진 사람만 다음 행동을 입력한다. 사건 발생 시 전원 즉시 소집. 무행동 자동 스킵. 전원 행동불능+가망 없음이면 자동 배드엔딩.
- **맵/통신(#180)**: 통신 1건이 도달하기까지 **차단→지연→수집→왜곡** 4단계 파이프라인을 구역·통로·매체·괴담수집범위 축으로 판정. 확실한 건 코드가, 애매한 것만 GM이, 비용 상한 시 전부 코드. @전체는 다음 턴 지연 출력 + 수신자 선별. 새 통신수단은 GM이 괴담 수집가능/차단 판정. 방문·인접 구역만 자동 등록.

핵심 원칙: **지킬 수 없는 약속 금지(엔진 근사 명시)**, **메타 비노출**, **저급 모델도 작동/상급 모델은 영화처럼**, **코드 우선·GM은 애매한 판정만**(토큰비용 억제).

## 1. 현행 진단

- **턴 진행**: `TRPGGameManager` + `TurnManager(turnMan)`. 행동은 채팅 입력 → AI 판정 → 서술 큐(`narrativeDelivery`)로 페이스드 출력. 무행동 워치독(#12)·자동스킵(#163 미구현)·기절/조종 스킵 존재. 시계는 `GameStateManager.tickClock`이 턴당 `minutes_per_turn` 고정 진행. → **행동별 소요시간·비동기 개념 없음**(모두가 사실상 동시 1턴).
- **시계**: `GameStateManager` — `start_time`~`end_time`, `minutes_per_turn`(고정), `time_visible`, `timeline.main_events`(time/deadline/branches), `advanceTimeline`, `skipTime`, `<TIME_SKIP>`. → 가변 완급 없음(항상 고정 분/턴).
- **통신**: `deliverDirectMessage`(1:1), `deliverPlayerBroadcast`(@전체 범위 게이트=위험·국지 시 같은 area), `broadcastToKnownContacts`(아는 번호 플레이어만), `handleNpcDirectComm`, `enqueueDelivery`/`processPendingDeliveries`(편지 지연), `entityInterferes`/`commModality`/`tamperText`(변조). `MapManager.sameArea`(보수적 area 판정). → **차단은 area/위험 위주, 구역별 봉쇄·통로봉쇄·매체별 지연·수집범위·새수단 판정 없음.** @전체는 즉시 전달(다음턴 지연·선별 없음).
- **맵**: `MapManager` — `zones/area/connections/distances`, `visitedZones`, `visibleAreas/knownAreaNames`(방문·현재만 노출=자동등록 부분구현), `sameArea`. `.gdam constraints.gated_zones`(정적), `comm_media`. → 자동등록은 area 단위로 이미 있음. 런타임 구역 봉쇄·매체 상태 없음.
- **상태·종료**: `status`(normal/puppet/dead), `faintTurnsRemaining`, `puppetRecoveryTurns/puppetTotalTurns/puppetGraceTurns`(방금 #1 추가), `onClearEnding`/게임오버/`stopSession`/`concludeWithReveal`. → **전원 행동불능 자동종료(#2) 없음**(전원 사망만 게임오버).

## 2. 턴 모델 설계

### 2.1 데이터 모델 (PlayerData)
```
long   busyUntilMin      // 이 인물이 자유로워지는 '게임 내 분'(절대). <=현재분이면 행동가능.
String currentActionText // 진행 중 행동 요약(busy 중 GM 짧은 서술·피격 판정 근거)
int    actionStartMin    // 행동 시작 분(경과 계산)
```
전역(GameStateManager): `long clockMin`(현재 게임 분, start_time 기준 누적) — 기존 tickClock을 분 누적으로 승격.

### 2.2 엔진 흐름 (핵심)
1. **행동 소요시간**: 플레이어 행동 입력 → GM 판정 응답에 `<DUR n>`(분) 태그. 없으면 코드 기본값(대화·관찰 2~5분 / 이동 = `zones.distances × 분당계수` / 대량작업·전투 GM 지정, 기본 10~30분). `busyUntilMin = clockMin + dur`.
2. **비동기 진행(스킵)**: "다음 턴"은 없다. 자유로운(busyUntil≤clockMin) 사람만 입력 가능. 전원이 busy면 시계를 **가장 이른 busyUntil**로 점프(그 사이 사건·타임라인 체크). = "아직 행동 중인 사람은 스킵, 남은 사람이 먼저 움직임".
3. **무행동 자동 스킵(#163)**: 자유로운 사람이 N초(실시간) 무입력이면 짧은 "관망/대기"(dur=최소)로 자동 처리 → 시계 진행. 기존 워치독 재활용.
4. **완급(중요순간 느리게)**: 전투·조우·타임라인 사건이 활성이면 `minutesPerAction` 가중치↓(같은 행동이 더 적은 분 소모=상대적으로 여러 행동 가능). GM이 `<PACE slow|normal|fast>`로 조절, 코드가 사건 태그(combat/근접조우)로 자동 slow.
5. **사건 즉시 소집**: 타임라인 main_event 발생·전투 개시·피격 시 `<SUMMON>` → 전원 `busyUntil=clockMin`(즉시 자유), 진행 중이던 긴 행동은 "중단/재개" GM 판정(§2.3).
6. **busy 중 처리**: busy인 사람도 (a)GM이 주기적으로 짧게 서술(내 다음 자유시점까지 K분 이상이면 중간 서술 1회), (b)피격 대상 가능, (c)통화 수신 가능(도착은 통신 파이프라인). 입력만 잠금.
7. **전원 행동불능 자동종료(#2)**: 매 시계진행 시 "살아있고 행동가능해질 수 있는 인원"이 0이고(전원 기절+자연회복 없음/조종 지속/사망), **탈출·클리어 경로가 닫힘**이면 → 배드엔딩(`concludeWithReveal`). 오종료 방지: ①기절/조종 자연회복 턴이 남아있으면 대기 ②아군 회복수단 보유자가 회복가능하면 대기 ③K회 연속 "전원 불능" 확인 후에만.

### 2.3 busy 중단/재개
`<SUMMON>`(피격·사건)으로 소집되면 긴 행동은 GM이 `interrupt|continue` 판정: 중단이면 부분효과+새 국면, 계속이면 다음 자유시점 유지. 기본은 위급도 높으면 중단.

### 2.4 시계·이동 연동
- 이동 소요 = `distances[from→to]`(칸/분) 사용. 없으면 인접=기본분. 원격 구역은 경로 합.
- `time_visible=false`면 스코어보드 시각 숨김 유지(분은 내부만).

### 2.5 GM 프롬프트/태그 (신설)
- `<DUR n>` 행동 소요 분. `<PACE slow|normal|fast>` 완급. `<SUMMON reason>` 즉시 소집. `<BUSY name n>` NPC/타인 장기행동.
- 프롬프트 지침: "행동마다 현실적 소요시간을 분으로 매겨라(<DUR>). 사소한 관찰·대화는 짧게, 이동·수리·의식은 길게. 이미 §페이싱: 결정적 단일행동은 1회로 완료(늘이지 마라)와 정합." (이미 넣은 페이싱 지침과 연결.)

### 2.6 스키마
`.gdam timeline`에 `action_costs`(선택: 유형별 기본 분)·`pace_events`(어느 main_event가 slow인지). 없으면 코드 기본.

## 3. 맵/통신 게이팅 설계 (#180)

### 3.1 데이터 모델
- Zone 런타임 상태(맵): `Map<zoneId, ZoneComm>` — `blockedIn/blockedOut`(지역봉쇄 방향별), `Set<passageBlocked>`(통로봉쇄=connection별), `mediumDelay{voice,text,signal,electronic,psychic → 분}`, `collectByEntity`(괴담 수집범위 여부).
- 이 상태는 GM 태그(`<ZONE_BLOCK zone dir>` `<PASSAGE_BLOCK a b>` `<COMM_DELAY zone medium n>` `<ENTITY_COLLECT zone on|off>`)로 런타임 설정. .gdam `constraints.gated_zones`는 초기값.

### 3.2 통신 1건 판정 파이프라인 (차단→지연→수집→왜곡)
`resolveComm(sender, target/scope, medium, msg)`:
1. **차단(코드 우선)**: 발신 구역 blockedOut / 수신 구역 blockedIn / 경로상 passageBlocked / 매체 수단 부재 → **불가**(코드 확실). "애매"(다른 지역인데 근처 등)만 GM 1회 판정.
2. **지연**: `mediumDelay[medium]` + 구역거리 → 도착 지연(분). 즉시=0. 지연분을 통신 파이프라인 시간으로.
3. **수집(엿듣기)**: 발신/수신/경로 구역 중 `collectByEntity`면 괴담이 내용 수집(`noteEntityIntel` 강도↑). 이미 있는 `entityInterferes`(매체 타입)와 AND.
4. **왜곡**: `entityInterferes(medium)` && 수집됨 && 확률 → `tamperText`(방금 #6에서 자연스러운 끊김/뒤집기로 개선).

### 3.3 @전체 지연큐 (#109)
- @전체는 **즉시 출력 금지 → 다음 시계진행에 도착**. 수신자 선별: 코드가 확실히 아는 수신불가자(매체 락·봉쇄·수단부재) 선제외 → **애매한 후보만 GM**이 "닿나?" 판정 → 후보 수×비용이 상한 초과면 **전부 코드 규칙**(같은 area+매체 가용)으로 폴백. NPC·괴담도 후보에 포함하되 매체 락/수집조건 볼 때만 반응(비용상 기본 제외 유지 가능 — 토글).

### 3.4 새 통신수단 GM 판정
플레이어가 taxonomy 밖 수단(예: 특수 장치·의식) 선언 → GM 1회 판정: 이 괴담이 그 매체를 수집/차단할 수 있나? → 결과를 ZoneComm/매체락에 반영(필드에 제한/추가 = 사용자가 말한 "필드에 제한을 걸거나 추가"). = #177에서 미룬 "새 방식 GM 판정"의 실제 구현 위치.

### 3.5 자동 구역 등록
현행 `visitedZones`/`visibleAreas`(방문·현재 area만 노출)을 유지·확장: 방문 OR 인접(연결된) 구역까지 "안다"로. 안 가본 먼 구역은 지도·선택지에서 숨김(이미 knownAreaNames가 부분 구현 — zone 단위로 확장).

### 3.6 NPC·괴담 매체락 반응
@전체·통신이 도달할 때 NPC/괴담은 자기 구역의 매체락/수집조건을 보고 반응 여부 결정(코드가 후보 필터 → 관련 NPC만 1회 AI). 무관 NPC는 값싼 정적 처리.

## 4. 두 시스템 상호작용
- **busy 중 통신 도달**: busy여도 통신은 수신(입력만 잠금). 지연 통신은 `busyUntil`과 무관하게 도착 시각에 파이프라인 처리.
- **이동 소요 + 구역 통신차단**: 이동 중(busy) 구역 전이는 도착 시 확정 → 통신 차단/수집 판정은 "현재 확정 구역" 기준. 이동 중 피격 소집(§2.3)과 정합.
- **완급 + 지연**: 중요순간(slow pace)엔 통신 지연분도 상대적으로 길게 체감(분 고정, 행동은 촘촘).

## 5. 단계적 구현 순서 (저위험→고위험, 작은 PR 단위)
1. **[저]** 시계 분 누적화(`clockMin`) + `<DUR>` 파싱(없으면 기본) — 표시/기록만, 스킵 미적용. 회귀 위험 낮음.
2. **[저]** 전원 행동불능 자동종료(#2) — 보수적 게이트(K회 확인·회복가망 체크). 종료 경로는 기존 `concludeWithReveal` 재사용.
3. **[중]** 무행동 자동 스킵(#163) — 워치독 재활용.
4. **[중]** 통신 파이프라인 `resolveComm` 도입(차단·지연만; 수집·왜곡은 기존 재사용). 구역봉쇄 태그.
5. **[중]** @전체 지연큐(#109) — 코드선별+애매만 GM+비용폴백.
6. **[고]** 비동기 busy 턴 본체(입력 게이팅·시계 점프·소집·busy중단재개) — 코어 루프 재구조화. ★가장 위험.
7. **[고]** 완급(pace) + 새통신수단 GM판정 + 매체락 반응.

## 6. 위험 부분 (승인 필요)

1. **비동기 busy 턴 본체(§2.2, 순서6)** — 현행 "동시 1턴" 루프를 근본 재구조화. 서술 큐·워치독·재도전/세이브·뷰어 재생 전반과 얽혀 회귀 위험 최고. → 단계1(시계·<DUR> 표시만)을 먼저 안정화 후, 별도 대형 PR로.
2. **전원 행동불능 자동종료(#2)** — 오종료 시 진행 중 세션을 파괴. 게이트(회복가망·K회 확인) 필수, 실플레이 검증 없이는 켜지 말 것(플래그로).
3. **@전체 GM 선별 비용(#109)** — 후보 많으면 토큰 폭증. 반드시 비용 상한→전부 코드 폴백을 먼저.
4. **구역/통로 봉쇄 상태의 세이브·재도전 지속성** — 런타임 ZoneComm를 저장/복원 안 하면 이어하기 시 유실. 세이브 스키마 확장 필요.
5. **NPC·괴담 @전체 반응(비용)** — 기본 제외 유지(사용자 기존 결정: 전체는 NPC 미수신). 켜면 비용↑ — 토글로만.
