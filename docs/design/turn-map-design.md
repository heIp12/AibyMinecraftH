# 턴(#151) · 맵/통신(#180) 완전 설계

작성: Opus 4.8 인라인 설계(멀티에이전트 울트라코드는 헤드리스 자율 실행에서 권한 스트림 미동작으로 미사용).
목적: 사용자가 확정한 턴/맵 개념을 현행 코드 기준으로 구현 가능한 설계로 정리. **구현은 승인 후** — §6 위험부분 참고.

---

## 0. 요약

두 시스템을 함께 설계한다.

- **턴(#151)**: 고정 "전원 1턴" 방식을 폐기하고 **행동별 소요시간(분) 기반 비동기 진행**으로. 각 플레이어는 자기 행동이 끝나는 시각(`busyUntil`)까지 "행동 중"이고, GM이 행동마다 소요시간을 매긴다. 시계가 가장 이른 `busyUntil`까지 흐르고, 그 시점에 자유로워진 사람만 다음 행동을 입력한다. 사건 발생 시 전원 즉시 소집. 무행동 자동 스킵. 전원 행동불능+가망 없음이면 자동 배드엔딩.
- **맵/통신(#180)**: 통신 1건이 도달하기까지 **차단→지연→수집→왜곡** 4단계 파이프라인을 구역·통로·매체·괴담수집범위 축으로 판정. 확실한 건 코드가, 애매한 것만 GM이, 비용 상한 시 전부 코드. @전체는 다음 턴 지연 출력 + 수신자 선별. 새 통신수단은 GM이 괴담 수집가능/차단 판정. 방문·인접 구역만 자동 등록.

핵심 원칙: **지킬 수 없는 약속 금지(엔진 근사 명시)**, **메타 비노출**, **저급 모델도 작동/상급 모델은 영화처럼**, **코드 우선·GM은 애매한 판정만**(토큰비용 억제).

## 1. 현행 진단

- **턴 진행**: `TRPGGameManager` + `TurnManager(turnMan)`. 행동은 채팅 입력 → AI 판정 → 서술 큐(`narrativeDelivery`)로 페이스드 출력. 무행동 워치독(#12)·자동스킵(#163 ★옵트인 구현★: `/trpg setting autoskip on` — 행동가능 전원 행동 시 즉시 한 걸음, turnMode 0/1, 기본 off)·기절/조종 스킵 존재. 시계는 `GameStateManager.tickClock`이 턴당 `minutes_per_turn` 고정 진행. → **행동별 소요시간·비동기 개념 없음**(모두가 사실상 동시 1턴).
- **시계**: `GameStateManager` — `start_time`~`end_time`, `minutes_per_turn`(고정), `time_visible`, `timeline.main_events`(time/deadline/branches), `advanceTimeline`, `skipTime`, `<TIME_SKIP>`. → 가변 완급 없음(항상 고정 분/턴).
- **통신**: `deliverDirectMessage`(1:1), `deliverPlayerBroadcast`(@전체 범위 게이트=위험·국지 시 같은 area), `broadcastToKnownContacts`(아는 번호 플레이어만), `handleNpcDirectComm`, `enqueueDelivery`/`processPendingDeliveries`(편지 지연), `entityInterferes`/`commModality`/`tamperText`(변조). `MapManager.sameArea`(보수적 area 판정). → **차단은 area/위험 위주, 구역별 봉쇄·통로봉쇄·매체별 지연·수집범위·새수단 판정 없음.** @전체는 즉시 전달(다음턴 지연·선별 없음).
- **맵**: `MapManager` — `zones/area/connections/distances`, `visitedZones`, `visibleAreas/knownAreaNames`(방문·현재만 노출=자동등록 부분구현), `sameArea`. `.gdam constraints.gated_zones`(정적), `comm_media`. → 자동등록은 area 단위로 이미 있음. 런타임 구역 봉쇄·매체 상태 없음.
- **상태·종료**: `status`(normal/puppet/dead), `faintTurnsRemaining`, `puppetRecoveryTurns/puppetTotalTurns/puppetGraceTurns`(방금 #1 추가), `onClearEnding`/게임오버/`stopSession`/`concludeWithReveal`. → **전원 행동불능 자동종료(#2) 없음**(전원 사망만 게임오버).

## 2. 턴 모델 설계

### 2.1 데이터 모델 (PlayerData)
```
long   busyUntilMin      // 이 인물이 자유로워지는 '게임 내 분'(절대). <=현재분이면 행동가능.
String currentActionText // 진행 중 행동 요약(busy 중 GM 짧은 서술·피격 판정 근거)
int    actionStartMin    // 행동 시작 분(경과 계산)
```
전역(GameStateManager): `long clockMin`(현재 게임 분, start_time 기준 누적) — 기존 tickClock을 분 누적으로 승격.

### 2.2 엔진 흐름 (핵심)
1. **행동 소요시간**: 플레이어 행동 입력 → GM 판정 응답에 `<DUR n>`(분) 태그. 없으면 코드 기본값(대화·관찰 2~5분 / 이동 = `zones.distances × 분당계수` / 대량작업·전투 GM 지정, 기본 10~30분). `busyUntilMin = clockMin + dur`.
2. **비동기 진행(스킵)**: "다음 턴"은 없다. 자유로운(busyUntil≤clockMin) 사람만 입력 가능. 전원이 busy면 시계를 **가장 이른 busyUntil**로 점프(그 사이 사건·타임라인 체크). = "아직 행동 중인 사람은 스킵, 남은 사람이 먼저 움직임".
3. **무행동 자동 스킵(#163)**: 자유로운 사람이 N초(실시간) 무입력이면 짧은 "관망/대기"(dur=최소)로 자동 처리 → 시계 진행. 기존 워치독 재활용.
4. **완급(중요순간 느리게)**: 전투·조우·타임라인 사건이 활성이면 `minutesPerAction` 가중치↓(같은 행동이 더 적은 분 소모=상대적으로 여러 행동 가능). GM이 `<PACE slow|normal|fast>`로 조절, 코드가 사건 태그(combat/근접조우)로 자동 slow.
5. **사건 즉시 소집**: 타임라인 main_event 발생·전투 개시·피격 시 `<SUMMON>` → 전원 `busyUntil=clockMin`(즉시 자유), 진행 중이던 긴 행동은 "중단/재개" GM 판정(§2.3).
6. **busy 중 처리**: busy인 사람도 (a)GM이 주기적으로 짧게 서술(내 다음 자유시점까지 K분 이상이면 중간 서술 1회), (b)피격 대상 가능, (c)통화 수신 가능(도착은 통신 파이프라인). 입력만 잠금.
7. **자동 배드엔딩(#2) — 두 경로**: 매 시계진행 시 아래 중 하나면 → 배드엔딩(`concludeWithReveal`).
   - (A) **전원 행동불능**: "살아있고 행동가능해질 수 있는 인원"이 0(전원 기절+자연회복 없음/조종 지속/사망) + 탈출·클리어 경로 닫힘.
   - (B) **★전원 생존이어도 도주 가망조차 없음★**(사용자 추가): 전원 살아 있어도 탈출·해결·생존 경로가 모두 닫혀 무엇을 해도 파멸뿐인 상태(현행 GM 프롬프트 '절망 가속' 274~277과 동일 판단을 엔진이 보조). 판단은 GM이 '가망 없음'을 선언(태그)할 때만 엔진이 종결을 확정 — 코드 단독 자동판정은 오종료 위험이 커 지양.
   - ★반드시 '끝 암시 → 마지막 기회 → 결말' 순서★(사용자 추가): 배드엔딩 직전 GM이 절망이 다가옴을 서술로 먼저 암시하고, 플레이어에게 마지막 발버둥(도주·희생·역전)의 1턴을 준 뒤 매듭짓는다. 아무 예고 없는 즉종료 금지. (이 완충 1턴은 오종료 방지 게이트도 겸한다 — 진짜 가망 없으면 그 턴에도 상황이 안 바뀐다.)
   - 오종료 방지: ①기절/조종 자연회복 턴이 남아있으면 대기 ②아군 회복수단 보유자가 회복가능하면 대기 ③미시도 길·아군·능력·탈출구가 하나라도 남으면 '가망 없음' 아님 ④'끝 암시' 완충 1턴 필수 ⑤K회 연속 확인 후에만.
   - **구현 완료(#2)**: 플래그 `AUTO_BADEND_ENABLED`(true, 문제 시 즉시 off). (A) 워치독 — 전원 무력화+회복가망 0이 K틱(≈30초) 연속이면 `onBadEnding`(전원 동물·완전조종 무한 틱도 매듭). (B) GM `<NO_HOPE/>` 태그(AiManager.parseNoHope+strip) — 연속 2회 확인+회복대기 게이트(anyRecoveryPending) 통과 시 `onBadEnding`, 선언 끊기면 리셋. 게이트 ②③은 GM 프롬프트(미시도 능력·길 남으면 선언 금지), ④⑤는 엔진 스트릭. onBadEnding 서술은 '왜 졌는지 납득' 지침. 리셋 3곳에서 카운터 초기화. 미결(테스트): 실플레이로 발동 빈도·오종료 검증(문제 시 K↑ 또는 플래그 off).

### 2.3 busy 중단/재개
`<SUMMON>`(피격·사건)으로 소집되면 긴 행동은 GM이 `interrupt|continue` 판정: 중단이면 부분효과+새 국면, 계속이면 다음 자유시점 유지. 기본은 위급도 높으면 중단.

### 2.4 이동 모델 — ★발상 뒤집기(플레이어 선택 → GM은 막을 때만)★ (사용자 확정)
현행: 플레이어 선언 → GM이 `<ZONE_UPDATE>`를 **내야만** 이동(옵트-인). 실패모드 = 태그 누락→제자리(desync), 틀린 zone_id→위치오염.
개편: **플레이어가 목적지를 선택 → 엔진이 기본으로 이동시킴(결정론적) → GM은 상황상 막을 때만 개입. 못 막으면 이동 성립(옵트-아웃).**

1. **선택 범위 = ★아는 모든 구역★(인접 한정 아님)**: `visitedZones`+known 전체를 목적지 후보로 제시(미발견·미지 구역 제외 — 스포/환각 차단). 검증된 목록이라 zone_id 환각 불가.
2. **경로(BFS)**: 목적지가 비인접이면 인접 그래프(`connections`)로 최단 경로 계산 → 홉 단위로 통과. 예: 집→마트 직행 ✗, 집→밖→시장→마트 ○. **위치는 홉마다 갱신**(각 구역을 실제로 지나며 그곳의 조우·사건·편지 등을 만난다).
3. **턴/시간 소모 = 경로 길이만큼**: 현행 고정턴 모델에선 **1홉 = 1턴**(traveling 상태로 자동 진행, 그동안 그 플레이어의 입력은 '이동 중'으로 스킵). #151 도입 시 `distances[from→to]×분당계수`로 정밀화(원격=경로 합).
4. **하드 차단(코드, 결정론적)**: 잠긴 구역(열쇠 없음)·경로 없음 → 애초에 후보에서 제외. 경로 중간이 잠겨 있으면 그 앞 구역에서 정지(도착 실패 알림).
5. **소프트 차단(GM, 상황적)**: `<BLOCK_MOVE player="X" reason="…">` — 괴담이 돌아서는 붙잡음·복도 붕괴 등 극적 순간만. **기본값=이동 성립**, 차단은 드물게. (낙관적 이동+거부권: 엔진 선(先)이동 + GM 통지 → GM이 서술하며 필요 시 BLOCK_MOVE로 그 홉을 취소·정지.)
6. **★홉별 서술(사용자 확정 — 배치)★**: 이동 중 각 홉에서 ★한눈에 들어오는 것★만 간결히. **① 그 턴에 다른 플레이어가 행동하면 그 서술과 ★함께★ 묶어 서술 / ② 아무도 행동 안 하면(이동만) 홉마다 장황히 쓰지 말고 ★이동이 끝난 뒤 종합 서술★**(경유지에서 눈에 띈 것 요약 + 도착지 묘사).
7. **중단/재개**: 이동 중 피격·`<SUMMON>` 시 traveling 일시정지 → GM `interrupt|continue`(§2.3과 동일). 중단이면 현재 홉 구역에 멈춤.
8. **자유서술 병행**: 채팅 "북쪽으로 간다"도 계속 파싱 — GM이 목적지로 해석해 같은 엔진 경로(선택기와 동일 처리)로. 선택기 = 신뢰 주경로, 자유서술 = 몰입 보조.
9. `time_visible=false`면 스코어보드 시각 숨김 유지(분은 내부만).

**데이터(PlayerData)**: `List<String> travelPath`(남은 홉 큐), `String travelDest`(최종 목적지 표시). 위치=`zone`(홉마다 갱신). 세이브 포함.
**단계 구현**: (선행·저위험) BFS 경로 helper + 아는-구역 이동 선택기 + `<BLOCK_MOVE>` 파싱/스트립 + travelPath 필드 → (턴결합) traveling 자동 진행(1홉/턴)·홉별 서술 배치. 턴루프 결합부는 #151 busy 모델과 정합.
**구현 완료(#190)**: `MapManager.shortestZonePath`(BFS) · `PlayerData.travelPath/travelDest/isTraveling` · `AiManager.parseBlockMoveTags`(+스트립) · `TRPGGameManager.advanceOneHop`(단일 커밋 지점)·`travelTurn`·`startTravel`·`openMoveSelector` · `/trpg 이동` 커맨드 · handleGameChat 이동게이트+동반전진 · onGmResponse `<BLOCK_MOVE>` 소비 · PromptBuilder 이동모델 지침. 커밋 fca7b95·a7fb57e·d70d6d3.
**미결(테스트 필요 — 눈감고 금지)**: 중단/재개 GM 판정(§2.4-7 interrupt|continue, 현재는 피격시 advanceOneHop 가드로 정지만) · §2.4-6② '이동만' 종합서술 정밀형(현재는 홉별 간결서술) · 자유서술 이동해석(§2.4-8, 현재는 선택기 주경로만) · #151 distances×분당계수 정밀화(현재 1홉=1턴 고정).

### 2.4a 거대 영역(realm) — 걸어서 못 넘는 분리 세계 (사용자 확정)
현실↔꿈, 지구↔화성처럼 ★도보로 오갈 수 없는 완전 분리 세계★는 realm으로 구분한다. "이동거리·속도가 어떻게 되나"의 답 = **비물리 전이는 걷기가 아니라 GM 서술 사건**이다.
- **거리**: 없음(공간 인접 무의미) → BFS·이동 선택기에 안 뜬다. **속도/시간**: 걷는 홉이 아니라 트리거의 성격대로(잠들면 즉시 · 우주항행이면 `<TIME_SKIP>`).
- **모델**: zone에 `realm` 필드(기본 "" = 주 영역). `MapManager.realmOf` + `shortestZonePath`가 realm 경계를 넘지 않음 → 다른 realm은 도보 불가. realm 전이는 GM이 트리거로 `<ZONE_UPDATE>`(런타임 예외적 직접 이동).
- **왜곡**: 비현실 realm(꿈·이세계) 내부는 거리·시간이 뒤틀릴 수 있음 — realm 내부 도보는 시스템이 처리하되 GM이 서술로 초현실 감각(1분=몇 시간·비유클리드·제자리 등)을 덧입힌다. 왜곡이 이동을 실제로 막아야 하면 `<BLOCK_MOVE>`.
- **구현 완료(#190)**: `MapManager`(zoneRealm·realmOf·BFS realm 경계) · `PromptBuilder`(realm 전이·왜곡 지침) · `GdamGenerator`(zone realm 필드·연결 금지·초반 회피 지침). 초반 스테이지는 realm 분리 대신 '서술 근사' 권장(생성기 지침).

### 2.5 GM 프롬프트/태그 (신설)
- `<DUR n>` 행동 소요 분. `<PACE slow|normal|fast>` 완급. `<SUMMON reason>` 즉시 소집. `<BUSY name n>` NPC/타인 장기행동. `<BLOCK_MOVE player reason>` 이동 소프트 차단(§2.4-5 — 드물게·극적일 때만, 기본은 이동 성립).
- 프롬프트 지침: "행동마다 현실적 소요시간을 분으로 매겨라(<DUR>). 사소한 관찰·대화는 짧게, 이동·수리·의식은 길게. 이미 §페이싱: 결정적 단일행동은 1회로 완료(늘이지 마라)와 정합." (이미 넣은 페이싱 지침과 연결.)

### 2.6 스키마
`.gdam timeline`에 `action_costs`(선택: 유형별 기본 분)·`pace_events`(어느 main_event가 slow인지). 없으면 코드 기본.

## 3. 맵/통신 게이팅 설계 (#180)

### 3.1 데이터 모델
- Zone 런타임 상태(맵): `Map<zoneId, ZoneComm>` — `blockedIn/blockedOut`(지역봉쇄 방향별), `Set<passageBlocked>`(통로봉쇄=connection별), `mediumDelay{voice,text,signal,electronic,psychic → 분}`, `collectByEntity`(괴담 수집범위 여부).
- 이 상태는 GM 태그(`<ZONE_BLOCK zone dir>` `<PASSAGE_BLOCK a b>` `<COMM_DELAY zone medium n>` `<ENTITY_COLLECT zone on|off>`)로 런타임 설정. .gdam `constraints.gated_zones`는 초기값.

### 3.2 통신 1건 판정 파이프라인 (차단→지연→수집→왜곡)
`resolveComm(sender, target/scope, medium, msg)`:
1. **차단(코드 우선)**: 발신 구역 blockedOut / 수신 구역 blockedIn / 경로상 passageBlocked / 매체 수단 부재 → **불가**(코드 확실). "애매"(다른 지역인데 근처 등)만 GM 1회 판정.
2. **지연**: `mediumDelay[medium]` + 구역거리 → 도착 지연(분). 즉시=0. 지연분을 통신 파이프라인 시간으로.
3. **수집(엿듣기)**: 발신/수신/경로 구역 중 `collectByEntity`면 괴담이 내용 수집(`noteEntityIntel` 강도↑). 이미 있는 `entityInterferes`(매체 타입)와 AND.
4. **왜곡**: `entityInterferes(medium)` && 수집됨 && 확률 → `tamperText`(방금 #6에서 자연스러운 끊김/뒤집기로 개선).

### 3.3 @전체 지연큐 (#109)
- @전체는 **즉시 출력 금지 → 다음 시계진행에 도착**. 수신자 선별: 코드가 확실히 아는 수신불가자(매체 락·봉쇄·수단부재) 선제외 → **애매한 후보만 GM**이 "닿나?" 판정 → 후보 수×비용이 상한 초과면 **전부 코드 규칙**(같은 area+매체 가용)으로 폴백. NPC·괴담도 후보에 포함하되 매체 락/수집조건 볼 때만 반응(비용상 기본 제외 유지 가능 — 토글).

### 3.4 새 통신수단 GM 판정
플레이어가 taxonomy 밖 수단(예: 특수 장치·의식) 선언 → GM 1회 판정: 이 괴담이 그 매체를 수집/차단할 수 있나? → 결과를 ZoneComm/매체락에 반영(필드에 제한/추가 = 사용자가 말한 "필드에 제한을 걸거나 추가"). = #177에서 미룬 "새 방식 GM 판정"의 실제 구현 위치.

### 3.5 자동 구역 등록
현행 `visitedZones`/`visibleAreas`(방문·현재 area만 노출)을 유지·확장: 방문 OR 인접(연결된) 구역까지 "안다"로. 안 가본 먼 구역은 지도·선택지에서 숨김(이미 knownAreaNames가 부분 구현 — zone 단위로 확장).

### 3.6 NPC·괴담 매체락 반응
@전체·통신이 도달할 때 NPC/괴담은 자기 구역의 매체락/수집조건을 보고 반응 여부 결정(코드가 후보 필터 → 관련 NPC만 1회 AI). 무관 NPC는 값싼 정적 처리.

  - **층1(유휴/원거리)**: 비트 없는 못 닿는 NPC는 AI 호출 없이 예정만 GM 정적 주입(막후 진행). 유지.
  - **층3 능동 비트(핵심)**: 턴당 자율 구동을 ★소수만★.
    - (A) 활성 창 NPC = 매턴. `npcActiveUntil` 맵으로 관리: ①플레이어 지시·상호작용(handleNpcDirectComm)→now+3, ②NPC가 `<BUSY turns="N"/>`로 '다급한 일 중' 선언→now+min(5,N). 대화 쿨다운(직전 1턴)엔 자율 구동 안 함(맥락오염 방지). 활성은 최대 2명.
    - (B) 라운드로빈 1명 = ★주기 턴(cadence)에만★(N주기마다 1명, `npcBeatCursor` 순번). 대화 중 NPC는 건너뜀.
  - **능동 도달(빠졌던 핵심 의도)**: 라운드로빈이 ★못 닿는 NPC도★ 골라 프롬프트로 '목표가 닿는 것이면 스스로 이동/연락하라'(GM이 NPC_AT로 위치 이동, NPC_CALL로 원거리 연락). npcCanReachAnyPlayer 전면 스킵 폐기.
  - **역할 우선(계획)**: role_type이 자유서술(고정 enum 아님)이라 코드 분류 불가 → ★AI가 motivation으로 접근·연락 여부 자율 판단★(코드는 '누가 비트를 받나'만, NPC AI는 '접근할까'를 결정). 순수 라운드로빈+활성창으로 대체.
- **효율**: 활성 NPC(보통 1~2)만 매턴 + 라운드로빈은 주기 턴 1명 → 턴당 대략 1~3콜로 bound(임시의 '전원 매턴'보다 대폭↓, 고가치 순간에 집중).
- **미결(테스트 필요)**: 층2 사건창발(사건→관련 NPC 1회 정밀 매칭)은 미구현(라운드로빈이 근사) · role-우선 정밀화 · <BUSY> 남발 시 비용 · 실플레이로 페이스·비용 검증.
