# 단체턴(통합 라운드) 설계

작성: Opus 4.8 인라인 설계(멀티에이전트·대화형 질문 도구는 헤드리스 자율 실행에서 권한 스트림 미동작으로 미사용).
목적: 플레이어별 개별 GM 호출을 **행동가능 전원 행동 수집 후 GM 1회 통합 처리**(단체턴)로 바꾸는 설계.
**증분 1(config 토글·세팅 UI)은 구현·커밋 완료.** 이 문서는 증분 2~4(라운드 루프·NPC 배치·거리시간 이동)의 설계와 **핵심 미결정(서술 분배)**을 정리. 컴파일·실행 불가 환경이라 **증분 2 이후 구현은 서술 분배 방식 확정 후** 진행(테스트 가능 환경 권장).

관련: 턴 개편 #151, 맵/통신 #180 (`turn-map-design.md`), 비용 과다 #231.

---

## 0. 요약 · 동기

**왜 단체턴인가** (사용자 요청 근거):
1. **일관성**: 개별 호출은 A의 GM 호출과 B의 GM 호출이 서로 몰라, 같은 괴담을 각자 옆 구역에 등장시키면 한 괴담이 여러 구역에 복제된다. 통합 1회 호출은 판 전체를 한 번에 봐서 괴담을 한 번만 배치.
2. **비용 절감**: 개별 호출은 턴 컨텍스트(최근 사건·주변 상태)를 N번 중복해서 실어 보낸다(시스템 프롬프트만 캐시, 이 컨텍스트는 비캐시). 통합은 한 번만. #231(비용 40% 초과)의 유력 원인.
3. **TRPG 정석**: '전원 행동 뒤 GM이 한 번에 정리'는 테이블탑 TRPG의 기본 리듬.

**핵심 원칙**: `groupTurn=false`(개별턴)는 현행 동작 100% 보존(회귀 0). 기본값은 `true`(단체턴). 시계 축(`turnMode` 0/1/2)과 **별개 축**.

## 1. 현행 진단 (증분 2 착수 기준)

- **행동 처리**: `handleGameChat`(TRPGGameManager ~2188) → 많은 특수분기(밀담·꼭두각시·변신·기절·미등장·서술중·busy·디바운스·금지어·통신@·능력) 후, 평범한 행동은 `turnMan.handleAction(player, actionMessage, gmSystemPrompt, gmCtx)`(~2513)로 **즉시 개별 GM 호출**.
- **`TurnManager.handleAction`**(56~107): 플레이어 1명 → `ai.callGmAi(...)` 1회 → `responseHandler`(=`onGmResponse`)로 그 1명 응답 전달. 다른 IDLE 플레이어는 동시 입력 가능(개별·병렬).
- **`onGmResponse`**(~2561): ★단일 행동자(`response.player()`) 기준★으로 클리어·STATE_UPDATE·주사위·`deliverNarrative(player, raw)` 처리.
- **태그 파싱은 이미 다중 플레이어 대응**: `parseZoneUpdates`→`[{player,zone,...}]` 리스트(태그마다 플레이어명), STATE_UPDATE도 player 필드 키잉. → 여러 명 위치·상태 갱신은 한 응답에서 처리 가능.
- **장면 격리**: `GameStateManager.buildTurnInput`(871)이 같은 구역/다른 구역+거리로 분리, `isPerceivableEvent`로 다른 구역 행동을 서술 컨텍스트에서 배제. 플레이어는 **자기 구역 장면만** 본다.

**결론**: 태그 처리는 배치 가능하나, **서술 전달(`deliverNarrative`)이 단일 행동자용**이고 **장면 격리**가 유지된다. 통합 서술 1개를 어떻게 분배하느냐가 핵심.

## 2. ★핵심 미결정★ — 통합 서술 분배 (구현을 가름)

단체턴은 GM이 **하나의 통합 서술**을 낸다. 그런데 전원에게 그대로 뿌리면 다른 구역 장면이 새어 **장면 격리가 깨진다**. 세 안:

### 안 1. 구역별 분할 서술 ★권장★
GM이 통합 서술을 구역/인물 단위로 나눠 출력(예: `<SCENE zone="zone_C">…</SCENE>` 또는 `[구역명]` 헤더). 엔진이 각 조각을 **해당 구역 플레이어에게만** 전달.
- 장점: '한 번의 통합 호출'이라는 사용자 의도 그대로 + 장면 격리 유지 + 최대 일관성·비용절감.
- 단점: GM이 구역 태그를 신뢰성 있게 출력해야 함. 새 분배 파서 필요(테스트 필수). 소형 모델 준수도 리스크.
- 폴백: 구역 태그가 없으면 안 3(같은 구역 묶음)처럼 동작하도록 방어.

### 안 2. 전체 공유 (단순)
통합 서술을 라운드 참가 전원에게 그대로 전달.
- 장점: 구현 가장 쉬움(기존 delivery에 수신자만 확장).
- 단점: **장면 격리 붕괴** — 다른 구역 장면이 서로 보임(영화 관전 톤). 기존 비주얼/스포일러 모델과 충돌(뷰어 시점 가시성과도 어긋남). ★비권장★.

### 안 3. 같은 구역끼리만 배치 (안전 폴백)
한 구역에 모인 플레이어들만 묶어 통합 처리(구역당 GM 1회), 흩어진 플레이어는 개별.
- 장점: 기존 단일-장면 delivery 파이프라인 재사용(회귀 위험 최소) + 장면 격리 자연 유지 + 파티가 함께인 흔한 경우 비용·일관성 이득.
- 단점: 흩어져 있으면 개별턴과 동일(전역 배치 아님). 사용자의 '모든 플레이어' 의도와 부분적으로만 일치.
- 크로스구역 일관성(괴담 다중구역)은 **증분 1의 위치 권위 규칙**(커밋 `a80d923`)이 이미 완화.

**권장**: 안 1(구역별 분할)을 목표로, **안 3을 폴백**으로 내장(구역 태그 실패 시 자동 강등). 이러면 의도(1회 통합)·격리·안전을 모두 확보. 테스트 가능 환경에서 안 1 파서 검증 필수.

## 3. 라운드 배리어 설계 (증분 2)

**상태 추가** (TRPGGameManager):
- `Map<UUID,String> pendingRoundActions` — 이번 라운드 대기 행동(행동자→행동문).
- `Map<UUID,String> pendingRoundCtx` — 행동별 turnCtx(능력·주사위 프리롤 등).
- `boolean roundFlushing` — 플러시 중 재진입 차단.

**개입 지점**: `handleGameChat`의 평범한 행동 resolution(현 ~2513 `turnMan.handleAction`) 직전.
```
if (state.isGroupTurn() && <평범한 행동 경로>) {
    pendingRoundActions.put(uuid, actionMessage);
    pendingRoundCtx.put(uuid, gmCtx);
    player 알림("행동 접수 — 대기 N/M");
    maybeFlushRound(zone);   // 배리어 판정
    return;                  // 즉시 GM 호출 안 함
}
// groupTurn=false → 기존대로 즉시 handleAction (회귀 0)
```
⚠️ **개별 즉시 처리 유지 대상**(배치 금지): 통신(@전체·@NPC), 능력 발동, 밀담, 꼭두각시·변신·기절 유도 행동. 이들은 기존 즉시 경로 그대로(개입은 '평범한 행동'만).

**배리어 판정** `maybeFlushRound`:
- 대상 = (안 3) 그 구역의 행동가능(살아있음·등장·비busy·비무력) 플레이어 / (안 1) 전체 행동가능 플레이어.
- 전원 제출됐거나, **타임아웃**(현실 N초 무입력) 도달 시 플러시. #163 무행동 자동스킵·워치독 로직 재사용.
- 미제출자는 '행동 없음(관망)'으로 스킵(#163 정신 계승).

**플러시** `flushRound`:
1. 대기 행동 수집(+증분 3: NPC 행동).
2. 통합 turnInput 조립: `여러 인물이 동시에 행동한다:\n[이름A] 행동A\n[이름B] 행동B…` + 각자 turnCtx 병합 + (안 1) 구역 분할 서술 지시.
3. **GM 1회 호출**(`callGmAi`).
4. 응답 처리: 태그(ZONE_UPDATE·STATE_UPDATE)는 기존 다중대응 파싱 그대로. 서술은 §2 안에 따라 분배.
5. 큐 클리어, `roundFlushing=false`.

**주사위**: 다인 동시 판정은 인라인 주사위(#254)로 서술 중 판정 유도(별도 per-player DICE 태그 남발 방지). 필요 시 DICE 리스트 처리.

## 4. NPC 행동 배치 (증분 3)

"NPC도 GM에게 플레이어처럼 행동을 보내 단체턴 진행" (사용자 요청).
- 라운드 플러시 직전, 그 장면 관련 critical NPC AI가 자기 행동을 1문장 결정(기존 `fireEntityActorForTurn`·자율 NPC 발화 재사용) → 통합 입력에 `[NPC 이름] 행동`으로 플레이어와 함께 포함.
- 기존 자율 NPC/엔티티 호출을 **라운드 단위로 통합**(턴마다 흩어진 호출 → 라운드당 모아서). idle NPC는 정적 예정 주입(#179 3층화 정신 유지).
- 상한: 장면당 NPC 소수(비용). 닿는 플레이어 없으면 생략(기존 `entityCanReachAnyPlayer`).

## 5. 거리·시간 비례 다구역 이동 (증분 4)

"매 턴 한 구역이 아니라 거리·흘러간 시간에 비례해 한 번에 이동" (사용자 요청). 현재 선택기 이동은 매 턴 한 홉(`advanceOneHop`, #190).
- 라운드가 흘린 시간(Δ분) = 그 라운드 행동들의 소요시간(#151 DUR)로 산정.
- 각 이동 선언 플레이어는 Δ분 동안 갈 수 있는 만큼 `zones.distances`(도보 분)를 소진하며 **여러 홉 진행**(중간 구역 통과 서술은 요약).
- 잠금·봉쇄(#180)·realm 경계는 기존대로 정지. 남은 거리는 다음 라운드로 이월.
- `advanceOneHop`을 `advanceByMinutes(Δ)`로 확장(한 홉 → 시간예산 소진 다홉). turnMode 0(고정)에선 Δ=minutes_per_turn.

## 6. 리스크 & 단계별 구현

**리스크**:
- 핵심 게임 루프(9000줄 God class) 개입 → 회귀 위험. **`groupTurn` 게이트로 개별턴은 100% 보존**해 회귀면을 단체턴에 한정.
- 컴파일·실행 불가 → 배리어·분배는 실플레이 테스트 필수. **기본값 단체턴이라 버그 시 기본 경험이 깨짐** → 테스트 전까지는 개별턴으로 검증 후 단체턴 확대 권장.
- 서술 분배(§2 안 1) 파서가 GM 출력 형식에 의존 → 폴백(안 3) 반드시 내장.
- 클리어/엔딩 판정이 단일 행동자 기준(`onGmResponse`) → 통합 응답에서 '누가 클리어를 달성했나'(`by`) 처리 재확인.

**단계**(각 단계 bracecheck + 가능하면 실플레이):
1. (완료) 증분 1: config 토글·세팅 UI.
2. 증분 2a: 라운드 배리어 + 큐 + 플러시 뼈대(안 3 같은구역 배치부터 — 기존 delivery 재사용, 최저위험).
3. 증분 2b: 안 1 구역별 분할 서술(GM 지시 + 분배 파서 + 폴백).
4. 증분 3: NPC 행동 배치.
5. 증분 4: 거리·시간 다구역 이동.
6. #231 비용 실측 재검증(통합 후 절감 확인).

## 7. 승인/확정 대기 항목

- **§2 서술 분배 방식**: 권장=안 1(구역 분할)+안 3(폴백). 사용자 확정 시 그대로, 아니면 지정 안으로.
- **배리어 타임아웃**(현실 N초): 기본 제안 20~30초 + 전원 제출 시 즉시.
- **기본값**: 사용자 요청대로 단체턴 true. (테스트 전 잠정 개별턴 운용 여부는 운영 판단.)
